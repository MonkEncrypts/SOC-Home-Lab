# Detection Philosophy

The objective of this detection is not to identify every execution of `cmd.exe` or native Windows utilities. These binaries are commonly used by administrators, support engineers, and legitimate users during normal system operation.

Instead, the detection aims to identify command-line activity that demonstrates behavioural characteristics consistent with post-compromise host reconnaissance while providing analysts with sufficient context to perform an effective investigation.

---

# Detection Workflow

The following workflow illustrates how telemetry progresses through the monitoring pipeline before becoming available for investigation.

```text
Windows Command Execution
            │
            ▼
Sysmon Event ID 1 (Process Create)
            │
            ▼
Splunk Universal Forwarder
            │
            ▼
Splunk Enterprise
            │
            ▼
SOC Investigation
```

The workflow demonstrates the complete telemetry lifecycle from endpoint activity through SIEM ingestion and analyst investigation.

| Component | Purpose |
|----------|---------|
| Windows Endpoint | Generates process execution activity |
| Sysmon Event ID 1 | Records process creation telemetry |
| Splunk Universal Forwarder | Forwards telemetry to the SIEM |
| Splunk Enterprise | Stores and enables investigation of collected telemetry |

# Detection Strategy

The investigation relied on endpoint telemetry generated by Sysmon Process Create events.

Rather than detecting a single executable, the investigation focused on identifying a sequence of reconnaissance commands executed from an interactive Command Prompt session.

The following characteristics increase the investigative value of this activity:

- Multiple reconnaissance commands executed within a short time window.
- Parent-child relationship between `cmd.exe` and spawned utilities.
- Sequential execution of host discovery commands.
- Consistent user context across executed processes.
- Rich process metadata recorded by Sysmon Event ID 1.

---

# Investigation Query

The following SPL query was used as the starting point for the investigation.

```spl
index=main source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational"
```

The investigation was then narrowed by searching for specific command execution.

```spl
index=main source="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational"
"whoami"
```

These searches demonstrate the investigative workflow rather than serving as production-ready detection rules.

---

# Detection Opportunities

This investigation highlights several opportunities for improving detection coverage in a production SOC.

Potential improvements include:

- Detect multiple reconnaissance commands executed within a defined time window.
- Correlate command execution with interactive logon sessions.
- Identify command execution initiated by suspicious parent processes.
- Combine Sysmon Process Create telemetry with PowerShell logging for broader execution visibility.
- Generate alerts when several native discovery utilities execute sequentially.

These behavioural approaches reduce reliance on individual executables and provide greater resilience against attacker variations.

---

# Detection Limitations

This investigation was intentionally performed using XML string matching because automatic Sysmon XML field extraction had been deferred during the current phase of the project.

Although this approach supports effective investigations within the home lab environment, production deployments should implement structured field extraction to enable more efficient searching, correlation, and dashboard development.

---

# Detection Summary

This detection demonstrates how endpoint telemetry generated by Sysmon can be leveraged to identify Windows command-line reconnaissance activity within Splunk Enterprise.

Rather than relying solely on individual executable names, the investigation emphasizes behavioural analysis, process relationships, and command execution patterns to support more effective SOC investigations.
